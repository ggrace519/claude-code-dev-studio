# Innovation Proposals — Claude Code Dev Studio
*Generated 2026-06-12 · based on commit c1991b0*

## How this codebase stands today

This is a shipped, versioned (v0.7.3) distribution system for Claude Code prompt assets: 19 always-on agents + ~90 JIT skills, dual bash/PowerShell installers, a `ccds` dispatcher, a `catalog.json` index, a SHA256-verified release pipeline, and CI that runs structural verification plus ShellCheck/PSScriptAnalyzer. The ADR discipline is genuinely excellent — seven ADRs trace every architectural turn, and ADR-0007 (the 105-agent → 19-agent + skills collapse) is a sophisticated, correct read of Claude Code's composition rules.

Where it's ordinary: the *content* is verified only structurally (BOM, kebab-case, frontmatter fields) — nothing checks that what the agents and catalog *say* is true or that the routing descriptions actually route. Distribution is 1,400 lines of hand-rolled installer doing what Claude Code's native plugin marketplace now does for free. The playbook's signature feature — "phase gates are real" — is unenforced prose. And small drift has already crept in: a nonexistent model ID, two different repo URLs in install instructions, and a catalog that CI never regenerates.

## What the best in this space are doing

- **Native plugin marketplaces** are the standard distribution path since plugins went stable: a `.claude-plugin/marketplace.json` catalog of versioned plugins, each shipping skills/agents/hooks/MCP, with pinning, auto-update (`/plugin marketplace update`), git/npm sources, and per-plugin enablement. See [official docs](https://code.claude.com/docs/en/plugin-marketplaces) and [anthropics/claude-plugins-official](https://github.com/anthropics/claude-plugins-official).
- **[obra/superpowers](https://github.com/obra/superpowers)** — the most admired skill library — distributes via [its own marketplace](https://github.com/obra/superpowers-marketplace) and uses hooks + initial instructions to make its workflow *enforced*, not advisory.
- **Activation testing is emerging**: [cc-plugin-eval](https://github.com/sjnims/cc-plugin-eval) generates direct/paraphrase/negative scenarios per component and verifies activation via the Agent SDK; [promptfoo tests agent skills](https://www.promptfoo.dev/docs/integrations/agent-skill/) for routing, quality, and cost with golden datasets in CI. Community consensus is that [subagent auto-routing is unreliable](https://buildtolaunch.substack.com/p/claude-code-subagents-explained) and description quality is the lever.
- **Hooks as enforcement**: elite setups use SessionStart to inject project state and PreToolUse as a deterministic gate — "turn your quality standards from documentation into enforcement" ([production hook patterns](https://www.pixelmojo.io/blogs/claude-code-hooks-production-quality-ci-cd-patterns), [hooks reference](https://code.claude.com/docs/en/hooks)).
- **[wshobson/agents](https://github.com/wshobson/agents)** ships one marketplace to six harnesses (Claude Code, Codex, Cursor, OpenCode, Copilot, Gemini CLI) — skills are becoming a cross-tool format.

## Proposals (ranked)

### 1. Ship the playbook as a native Claude Code plugin marketplace
**Category:** architecture
**Impact 5 · Novelty 4 · Effort 3 · Fit 5**

**The idea.** Generate a `.claude-plugin/marketplace.json` plus one plugin per pack (`ccds-saas` = `saas-architect` agent + `saas-*` skills; `ccds-core` = 5 core agents + cross-cutting skills) from the existing source tree. Users get the entire playbook with `/plugin marketplace add ggrace519/claude-code-dev-studio` and `/plugin install ccds-saas` — versioning, pinning, updates, and per-project enablement handled by Claude Code itself. The pack→plugin mapping is 1:1 with the architecture you already have.

**Inspired by.** [Plugin marketplace docs](https://code.claude.com/docs/en/plugin-marketplaces); [obra/superpowers-marketplace](https://github.com/obra/superpowers-marketplace); [anthropics/claude-plugins-official](https://github.com/anthropics/claude-plugins-official).

**Implementation sketch.** New `scripts/build-marketplace.py` (sibling of `build-catalog.py`) that emits `dist-marketplace/` — or better, a checked-in `.claude-plugin/` tree regenerated by the script: for each pack, a plugin dir with `.claude-plugin/plugin.json`, `agents/<pack>-architect.md`, `skills/<skill>/SKILL.md` (copied), plus a `ccds-core` plugin. `marketplace.json` lists all 15 with `metadata.pluginRoot: "./plugins"`, versions sourced from `version.txt`. CI gets a freshness job (regenerate + `git diff --exit-code`). The ZIP installer remains for users who want the `ccds` CLI and global layout; README documents both paths.

**Effort.** 1–2 days. Main risk: plugin-installed skills are namespaced and enabled per-plugin, which changes the `sync-agents`/manifest mental model — document the two paths as parallel, don't try to unify yet.
**First step.** Write `scripts/build-marketplace.py` generating one pack (`saas`) + `marketplace.json`, and install it locally via `/plugin marketplace add <path>`.

### 2. Routing-eval harness — CI-test the 109 descriptions as a retrieval system
**Category:** wow / DX
**Impact 4 · Novelty 5 · Effort 3 · Fit 5**

**The idea.** Your own convention says "descriptions are the routing surface" — but the routing surface has zero tests. Treat routing as a retrieval problem: a golden dataset of realistic task prompts, each labeled with the agent/skill that *should* activate, evaluated in CI. A fast deterministic mode (TF-IDF/lexical match against descriptions) catches description overlap, near-duplicates, and regressions on every PR; an LLM mode (via `claude -p` with a routing prompt) measures real routing accuracy before releases. When someone edits a description, CI tells them if `dataplat-etl` now shadows `dataplat-streaming`.

**Inspired by.** [cc-plugin-eval](https://github.com/sjnims/cc-plugin-eval)'s scenario taxonomy (direct / paraphrase / negative); [promptfoo's golden-dataset routing tests](https://www.promptfoo.dev/docs/integrations/agent-skill/); IR evaluation transplanted to prompt routing.

**Implementation sketch.** `evals/routing/golden.json` — ~3 prompts per pack + negatives ("write a haiku" → no agent). `scripts/eval-routing.py` (stdlib-only): loads `catalog.json`, scores each golden prompt against every description, asserts expected-in-top-k, and reports a pairwise description-similarity matrix flagging pairs above a threshold. CI job runs deterministic mode; `--llm` flag shells out to `claude -p` for release-time accuracy runs. Output: pass/fail + a Markdown routing report.

**Effort.** 1 day for deterministic mode + dataset; LLM mode is +half a day. Risk: lexical scoring is a proxy — mitigate by framing deterministic mode as *ambiguity lint*, LLM mode as *accuracy measurement*.
**First step.** Write 10 golden prompts for `saas`/`ai`/`infra` and confirm TF-IDF top-1 matches expectations on the current catalog.

### 3. Playbook lint — verify the claims, not just the frontmatter
**Category:** DX / quick win
**Impact 4 · Novelty 3 · Effort 4 · Fit 5**

**The idea.** `verify-agents` checks files are *well-formed*; nothing checks they're *true*. A `lint-playbook.py` that cross-validates the system's own invariants: every skill named in an agent's "Skills you compose" manifest exists in `skills/`; every domain skill is referenced by its domain agent; `catalog.json` is exactly what `build-catalog.py` would regenerate (drift gate); all GitHub URLs point to one canonical repo; description style rules from CLAUDE.md (single sentence, no `<example>`, no literal `\n`); and a token budget assertion that the always-on layer stays under its advertised ~850-token budget.

**Inspired by.** Original — derived from observed drift: `skills/sync-agents/SKILL.md` says `519lab/...` while everything else says `ggrace519/...`; `deploy-checklist.md` pins `claude-haiku-4-6`, which is not a current model ID; catalog freshness is manual per CLAUDE.md.

**Implementation sketch.** `scripts/lint-playbook.py`, stdlib-only, same shape as `build-catalog.py`. Parses agent bodies for backtick-quoted `<pack>-*` skill references, diffs against `skills/` dirs; regenerates the catalog to a temp path and byte-compares; greps URL patterns; counts description tokens (~chars/4). New `lint` job in `ci.yml`; new `ccds lint` passthrough in the dispatchers. Found issues (URL, model ID) fixed in the same change.

**Effort.** Half a day. Risk: manifest parsing must tolerate prose around skill names — anchor on backtick-quoted names only.
**First step.** Run the cross-reference check; it already fails today on the two known drifts, which proves its value immediately.

### 4. Model aliases — stop pinning model IDs that rot
**Category:** quick win
**Impact 3 · Novelty 2 · Effort 5 · Fit 5**

**The idea.** All 19 agents pin dated model IDs (`claude-opus-4-7` ×16, `claude-sonnet-4-6` ×2, and `claude-haiku-4-6` ×1 — the latter doesn't appear to be a real public model). Claude Code agent frontmatter accepts tier aliases — `opus`, `sonnet`, `haiku`, `inherit` — that track the current best model in each tier automatically. Swap every pin for its alias and the library never ships a stale or invalid model reference again.

**Inspired by.** [Claude Code subagent docs](https://code.claude.com/docs/en/sub-agents) (model alias support); derived from the stale pins in the codebase.

**Implementation sketch.** `sed`-scale change across `.claude/agents/*.md` (`claude-opus-4-7` → `opus`, `claude-sonnet-4-6` → `sonnet`, `claude-haiku-4-6` → `haiku`), regenerate `catalog.json`, update the model-tiering note in CLAUDE.md, and add an allowed-values check to the Proposal-3 linter so a dated ID can't come back.

**Effort.** ~1 hour. Risk: none meaningful; aliases are the documented recommendation.
**First step.** The sed pass + catalog regen.

### 5. Phase-gate hooks plugin — make "phase gates are real" actually real
**Category:** wow / feature
**Impact 4 · Novelty 4 · Effort 3 · Fit 4**

**The idea.** The playbook's identity is its NIST-SSDF phase discipline, but nothing enforces it. Ship an optional `ccds-phases` plugin: a `.claude/phase` state file, a `SessionStart` hook that injects current phase + the last 3 ADR titles (automating the Context Refresh Protocol verbatim), and a `PreToolUse` hook that warns-or-blocks phase-inappropriate actions (deploy commands while in Implementation, source edits during Hardening sign-off). A `/phase` command advances the gate and appends the exit-criteria checklist to the conversation.

**Inspired by.** [Hook enforcement patterns](https://www.pixelmojo.io/blogs/claude-code-hooks-production-quality-ci-cd-patterns); obra/superpowers' "instructions that ensure the agent follows the workflow instead of improvising"; the playbook's own unenforced Phase Framework.

**Implementation sketch.** A plugin dir (composes with Proposal 1): `hooks/hooks.json` wiring `SessionStart` → `phase-context.sh` (cat `.claude/phase`, head `DECISIONS.md`) and `PreToolUse` (Bash matcher) → `phase-gate.sh` (deny-with-reason on a small command-pattern × phase table, advisory by default, `strict` mode opt-in). Plus `commands/phase.md`. Pure bash + jq-free JSON via heredocs, matching repo conventions.

**Effort.** 1–2 days including the pattern table. Risk: overblocking annoys users — default to advisory (`permissionDecision: "ask"`), strict opt-in.
**First step.** SessionStart context hook only — zero risk, immediate daily value.

### 6. `ccds doctor` — operationalize every install bug you've ever fixed
**Category:** ops / DX
**Impact 3 · Novelty 3 · Effort 4 · Fit 5**

**The idea.** The git history is a museum of environment bugs: BOM-broken frontmatter, trailing null bytes, CRLF in shipped scripts, null profile content, non-ASCII PS output. Each was fixed reactively. `ccds doctor` checks for the whole class proactively on any machine: BOM scan of installed agents/skills, CRLF scan of scripts, PATH resolution, catalog-vs-installed-tree consistency, CLAUDE.md marker-block integrity, version-vs-latest, WSL/NTFS permission quirks.

**Inspired by.** `brew doctor` / `flutter doctor`; derived from this repo's own fix history (commits 21e0432, 9c38fa3, e1a5ce8, 5bf07c5).

**Implementation sketch.** `cmd_doctor` in `bin/ccds.sh` + twin in `ccds.ps1`, each check a function emitting `OK/WARN/FAIL + remedy` lines, exit non-zero on FAIL. Reuses the BOM/frontmatter logic already in `verify-agents.sh`. Every future install-bug fix adds a check — turning the changelog's whack-a-mole into a ratchet.

**Effort.** 1 day for both shells. Risk: dual-implementation drift — keep checks table-driven so the two stay comparable.
**First step.** Bash version with the four highest-recurrence checks (BOM, CRLF, marker block, PATH).

### 7. Multi-harness export — publish the skill library beyond Claude Code
**Category:** feature / reach
**Impact 4 · Novelty 4 · Effort 2 · Fit 3**

**The idea.** The 90 skills are harness-agnostic domain expertise in a format (SKILL.md + frontmatter) that's becoming a cross-tool standard. An export step emitting Codex/Cursor/Gemini-CLI-compatible layouts (mostly path conventions + small frontmatter transforms) multiplies the addressable audience of the same content for little marginal work — relevant to the commercial-license ambition in ADR-0005.

**Inspired by.** [wshobson/agents](https://github.com/wshobson/agents) (one marketplace, six harnesses); the emerging Agent Skills format convergence.

**Implementation sketch.** `scripts/export-harness.py --target codex|cursor|gemini` mapping each SKILL.md into the target's expected tree; release workflow attaches per-harness ZIPs. Start with the one target you can actually test.

**Effort.** 2–3 days including verifying activation in at least one foreign harness. Risk: foreign-harness conventions shift; pin to documented formats and mark exports experimental.
**First step.** Hand-port `saas-billing` to one foreign harness and confirm it activates before automating.

## Killed ideas (and why)

- **Semantic pack matcher (embeddings for `sync-agents`)** — the 14-row signal table routes fine; embeddings add a dependency for no measured gap. Resume-driven.
- **Rewrite dispatcher in Go/Rust single binary** — bash+PS works, is shipped, and the audience has both. Rewrite in disguise.
- **Skill-usage telemetry** — real value, but privacy posture + plumbing outweigh it for a solo-maintained library; revisit if the marketplace lands and install count grows.
- **Auto-generated docs site from catalog.json** — marginal over the README + catalog; the marketplace UI (Proposal 1) gives browsing for free.
- **`ccds adr new` scaffolding command** — agents already draft ADRs via `playbook-conventions`; a CLI template adds little.
- **Bash/PS behavioral parity test harness** — genuinely useful but heavy; the linter (P3) + doctor (P6) cover the highest-bleed surface first.

## Suggested order of attack

Land the quick wins first — Proposal 4 (model aliases, an hour) and Proposal 3 (playbook lint) — because the linter immediately guards everything the other proposals touch, and it already catches two live bugs. Then build Proposal 1 (plugin marketplace), the strategic move: it modernizes distribution while the ZIP path keeps working, and Proposal 5's hooks plugin slots straight into it. Proposal 2 (routing evals) can proceed any time after 3, sharing its catalog-parsing plumbing; 6 and 7 are independent follow-ons.
